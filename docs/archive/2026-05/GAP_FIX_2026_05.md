## 5. Security posture — verified, not modified

Before making changes I confirmed:

- All nine Spring services have a `SecurityConfig` with a `JwtAuthenticationFilter`, stateless sessions, and a default-deny `anyRequest().authenticated()` rule. Public exemptions are tight (login/register/refresh, swagger, actuator, public website settings).
- Spring CORS is restricted to `localhost:3000`, `127.0.0.1:3000`, and `*.leraacademy.edu.vn`.
- The new `/api/users/me/*` endpoints use `@AuthenticationPrincipal AuthUser` rather than trusting any client-supplied user id.
- `/me/change-password` requires the current password when supplied; the current frontend at `frontend/app/dashboard/settings/page.tsx` does send it.
- High-privilege impersonation calls `accessGuard.assertSuperAdmin()` and writes a `WARN`-level audit log.

The `frontend/middleware.ts` deliberately leaves `/api/*` ungated at the edge — backend services own that gate. No change.

### Low priority — naming smell

- Two paths for the same resource: `/api/dropdown-options` vs `/api/system-settings/category/dropdown_*`. Pick one.
- `/api/cms-pages` vs `/api/cms-settings`. Pick one.
- `/api/approvals/*` (in-memory demo) vs the new `/api/users/{id}/approve|reject`. Two parallel approval systems; the new one is the keeper.

### Security hardening — recommended next sprint

- The `token` cookie is set client-side via `js-cookie`, so it cannot be `HttpOnly`. Move to server-set HttpOnly cookies (next-auth-style or a thin `/api/auth/session` route).
- No CSRF token. Spring's `csrf().disable()` is acceptable for stateless JWT carried in `Authorization: Bearer`, but if you ever switch to cookie-based auth you must re-enable CSRF.
- No refresh-token rotation. `apiFetch` simply logs the user out on 401. A `/api/auth/refresh` round-trip would be a small change.
- nginx CORS allowlist is a regex against `*.leraacademy.edu.vn` — add explicit prod domains before launch.

## E. Refresh-token rotation

| File | Change |
|------|--------|
| `backend/identity_service/.../controller/AuthController.java` | New `POST /api/auth/refresh` that takes `{ "refreshToken": "..." }`, validates it via the existing `JwtService`, and returns a new access token + a fresh refresh token. Marked `@PreAuthorize("permitAll()")` to override the class-level role check, matching the existing `SecurityConfig` permitAll for this path. Returns 401 on any signature/parse failure (no internal leakage). |
| `frontend/lib/api.ts` *(rewritten)* | `apiFetch` now reads a `refreshToken` cookie, and on a 401 calls `/api/auth/refresh` exactly once (with in-flight de-duplication so 5 parallel 401s only mint one new pair). On success it retries the original request transparently. On failure it falls through to the existing redirect-to-login path. Also dispatches `lera:error` on non-401 failures so the toast container shows a message. |
| `frontend/app/auth/login/page.tsx` | Persists the `refreshToken` cookie (7-day expiry) on successful login. |
| `frontend/app/dashboard/layout.tsx` | The Logout button also clears the `refreshToken` cookie. |

The cookie is still client-set (`js-cookie`), so it is NOT yet `HttpOnly` —
that remains a separate hardening sprint.

## F. Bonuses — operational + UX

- **Nginx rate limiting** — added to `gateway/nginx/nginx.conf`:
  - `limit_req_zone` `auth_zone` (10 r/s, burst 20) applied to `/api/auth/`.
  - `limit_req_zone` `api_zone` (60 r/s) declared and ready to apply to other
    locations as needed.
  - `limit_req_status 429`.
- **`/api/health` aggregator** — new `HealthAggregatorController` in
  `identity_service` that probes each upstream's `/actuator/health` in parallel
  with a 1.5s per-probe timeout and a 2s overall timeout, returning a single
  `{ status: "UP" | "DEGRADED", services: { ... }, ts }` JSON. Marked
  `permitAll()` and gateway-routed via the new `/api/health` location. Useful
  for an ops dashboard widget.
- **Frontend ErrorBoundary** — `frontend/components/ErrorBoundary.tsx` wraps
  the entire app under the root `app/layout.tsx`. Renders a friendly
  "Something went wrong" surface with **Try again** / **Reload** buttons
  instead of a blank screen on a render error.
- **Toast system** — `frontend/components/Toast.tsx` exposes
  `toast(msg, "info" | "success" | "warn" | "error")` and a `<ToastContainer/>`
  rendered once at root. Listens for `lera:error` events so all unexpected
  failures (apiFetch, ErrorBoundary) surface as a visible toast.

# Wave 4 — HttpOnly cookies + 17 more admin pages (2026-05-02)

## A. HttpOnly cookie migration (cross-cutting)

We removed the JWT from JS-readable storage on the browser without breaking
non-browser callers (mobile, postman, server-to-server scripts) that still
use `Authorization: Bearer`.

### Backend

| File | Change |
|------|--------|
| `identity_service/.../security/AuthCookies.java` *(new)* | Helper that sets / clears `token` and `refreshToken` cookies with `HttpOnly`, configurable `Secure` (`lera.cookies.secure`) and `SameSite=Lax` (configurable via `lera.cookies.same-site`). |
| `identity_service/.../service/JwtService.java` | Exposes `getAccessTokenSeconds()` / `getRefreshTokenSeconds()` so the cookie `Max-Age` matches the JWT lifetime. |
| `identity_service/.../controller/AuthController.java` | `POST /api/auth/login` and `POST /api/auth/refresh` now mirror the access + refresh tokens into HttpOnly cookies (in addition to the JSON body, for backward compatibility). New `POST /api/auth/logout` (permitAll) clears them. `/refresh` accepts the refresh token from either the body OR the `refreshToken` cookie. |
| `identity_service/.../config/SecurityConfig.java` | Permits `/api/auth/logout` alongside `/login`, `/register`, `/refresh`. |
| **All 9 services'** `security/JwtAuthenticationFilter.java` (`identity`, `academy`, `payment`, `payroll`, `attendance`, `connect`, `ai_gateway`, `rule_engine`, `social_media`) | Each filter now extracts the JWT either from `Authorization: Bearer ...` OR from the `token` cookie via a new `extractJwt(request)` helper. Same-shape change everywhere — completely backward compatible. |

### Frontend

| File | Change |
|------|--------|
| `frontend/lib/api.ts` | Rewritten. `apiFetch` always sends `credentials: "include"`; on 401 it calls `/api/auth/refresh` cookie-based (no body needed when the refresh cookie is set). The legacy `Cookies.get("token")` path is preserved as a fallback so older sessions still work. New `hasAuthSession()` helper checks both legacy `token` and the new `tokenSet` flag. |
| `frontend/lib/auth.ts` *(new)* | `logoutSession()` — POSTs `/api/auth/logout` with credentials, then wipes JS-readable hint cookies and redirects to login. |
| `frontend/app/auth/login/page.tsx` | `axios.post(..., { withCredentials: true })`; stops calling `Cookies.set("token" / "refreshToken")` (those are HttpOnly now); only drops a non-sensitive `tokenSet=1` flag for the SPA to know the user is signed in. |
| `frontend/app/dashboard/layout.tsx` | Logout button now calls `logoutSession()` instead of touching cookies directly. |

### Operational notes

- **Defaults are dev-friendly**: `lera.cookies.secure=false` and
  `same-site=Lax` so HTTP-on-localhost still works. **In production**,
  set `lera.cookies.secure=true` and (if cross-site) `same-site=None`.
- **Rollout safety**: existing sessions keep working because
  - all filters fall back to the bearer header,
  - `apiFetch` keeps reading `Cookies.get("token")` if it exists,
  - the legacy refresh-token flow still works.
  Once everyone has logged in once after the deploy, the JS-readable token
  cookie disappears organically.
